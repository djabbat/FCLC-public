# Федеральный Клинический Консорциум (FCLC)
## Приватная инфраструктура для обучения медицинского ИИ без передачи сырых данных пациентов

**Версия 16.0 — с полной интеграцией экспертной оценки, правового фундамента и операционной доработкой**

---

## Одна фраза

Объединяем медицинские данные клиник и фармацевтических компаний для обучения ИИ без передачи сырых данных пациентов, с измеримым вкладом каждого участника и прозрачным распределением выгод.

---

## Проблема

Данные есть у всех, но они:

- **в разных форматах** — HIS, EHR, PACS, LIS, неструктурированные записи
- **под разными юридическими ограничениями** — GDPR, национальные законы, внутренние политики
- **никто не хочет их отдавать** — риск репутационный, юридический, коммерческий

**Итог:** большие модели ИИ для медицины обучаются на узких датасетах, а реальные клинические данные остаются неиспользованными. Врачи продолжают работать без инструментов ИИ, которые могли бы быть созданы на основе уже существующих данных.

---

## Решение

Каждый участник разворачивает у себя **локальный узел**, который:

1.  **Подключается** к HIS/EHR/PACS через адаптеры (HL7/FHIR, собственные API)
2.  **Обезличивает данные** с гарантиями (удаление идентификаторов, квази-идентификаторы, дифференциальная приватность)
3.  **Нормализует** в единую схему (OMOP CDM)
4.  **Отправляет** только обезличенные агрегаты или обновления модели (градиенты, веса)

**Центральный оркестратор:**
- Собирает обновления моделей
- Агрегирует (федеративное усреднение с безопасной агрегацией)
- Ведёт аудит и учёт версий
- Рассчитывает вклад каждого участника

**Данные не покидают клинику — только обучающие сигналы.**

---

## Приватность — архитектурно, а не на словах

| Уровень | Механизм |
|---------|----------|
| 1 | Удаление прямых идентификаторов (имя, номер, адрес, точная дата) |
| 2 | Обобщение квази-идентификаторов (возрастные группы, редкие диагнозы → suppression) |
| 3 | Оценка риска повторной идентификации на уровне записи (k-анонимность, l-разнообразие) |
| 4 | **Дифференциальная приватность (ε=1.0 за раунд, δ=10⁻⁵, общий бюджет ε_total≤10.0, механизм Гаусса, учёт через Rényi DP)** |
| 5 | Безопасная агрегация (оркестратор видит только агрегированный результат) |

**Результат:** даже при компрометации оркестратора или перехвате трафика восстановить данные пациентов невозможно.

---

### Модель угроз и безопасность

**Модель угроз (threat model):**
- **Оркестратор:** honest-but-curious (честный, но любопытный) — следует протоколу, но может пытаться извлечь информацию из получаемых данных
- **Узлы:** до 25% узлов могут быть вредоносными (Byzantine) — отправлять некорректные обновления с целью отравления модели (data poisoning)
- **Внешний атакующий:** может перехватывать сетевой трафик

**Меры защиты:**
- **Безопасная агрегация (SecAgg+):** Реализация протокола **Secure Aggregation (SecAgg)** по спецификации Bonawitz et al. (2017) с использованием пороговых схем разделения секрета (Shamir's Secret Sharing). **Реализация:** Используется модифицированная версия протокола из библиотеки **OpenMined PySyft**, интегрированная с центральным оркестратором. Протокол гарантирует, что оркестратор получает только сумму обновлений от группы узлов, а индивидуальные градиенты остаются скрытыми, даже если до 1/3 узлов сговариваются. **План валидации:** Протокол будет протестирован в рамках Фазы 1 (Техническая разработка) на тестовых датасетах, включая стресс-тесты на устойчивость к сговору узлов и проверку корректности агрегации.
- **Дифференциальная приватность:** Гауссов шум с параметрами (ε=1.0, δ=10⁻⁵) добавляется к **агрегированным** обновлениям на стороне оркестратора **после выполнения SecAgg**. Это является основной защитой от атак восстановления градиентов (gradient inversion), делая восстановление исходных данных вычислительно невозможным.
- **Минимальный размер батча:** ≥32 записей для снижения риска gradient inversion attacks (Zhu et al. 2019).
- **Робастная агрегация:** Использование **Krum** или **Multi-Krum** в качестве алгоритма агрегации для устойчивости к до 25% вредоносных узлов (Byzantine tolerance). Для не-IID данных (статистической неоднородности) основной алгоритм обучения — **FedProx**, который добавляет регуляризационный член для управления дрейфом локальных моделей.
- **Репутационный скоринг:** узлы с аномальным поведением (например, градиенты с аномально высокой нормой) автоматически исключаются из обучения на данном раунде.

## Единая клиническая схема

Используется **OMOP Common Data Model** (Observational Medical Outcomes Partnership) — наиболее распространённый стандарт для наблюдательных исследований. Адаптация:

| Домен | Стандарт |
|-------|---------|
| Диагнозы | ICD-10/11 |
| Лаборатория | LOINC |
| Лекарства | ATC, RxNorm |
| Процедуры | SNOMED CT, CPT |
| Демография | обобщённые категории |

Локальный узел преобразует данные из HIS/EHR в OMOP на месте. Поддерживается **FHIR** как альтернативный входной формат.

## Измерение вклада

**Оценка вклада осуществляется через аппроксимированное значение Шепли (Federated Shapley Value) по методологии Wang et al. (Fed-SV, AAAI 2020).**

Для каждого участника вычисляется его маргинальный вклад в производительность глобальной модели. Для масштабируемости при 5–10 узлах используется метод Монте-Карло (100–200 итераций), что даёт приемлемую точность при вычислительных затратах, порядок которых составляет O(n² × M), где n — число узлов, M — число итераций. Для 10 узлов и 200 итераций дополнительные затраты времени составляют ~10 минут на раунд, что допустимо для асинхронного федеративного обучения.

**Преимущества подхода:**
- **Теоретическое обоснование:** единственная схема распределения, удовлетворяющая аксиомам эффективности, симметрии, линейности и нулевого игрока (Shapley 1953)
- **Защита от манипуляций:** маргинальный вклад сложно искусственно завысить без реального улучшения модели
- **Пропорциональность:** участник получает кредиты, строго соответствующие его реальному влиянию на качество модели
- **Устойчивость к доминированию крупных участников:** Shapley value оценивает именно маргинальный вклад, а не абсолютный объём данных

**Вычислительная реализация:**
- Оценка Shapley value выполняется центральным оркестратором после каждого раунда или эпохи
- Используется метод Монте-Карло с 100–200 выборками перестановок узлов
- **Альтернативный подход для масштабирования:** при росте числа узлов (>20) может использоваться аппроксимация через влияние на градиенты (influence functions) или стратифицированная выборка

Участники получают **кредиты вклада**, которые конвертируются в выгоды.

**Механизм защиты от безбилетников (free-rider):** доступ к глобальной модели и её новым версиям предоставляется только участникам, чей накопленный скоринг вклада превышает установленный порог (например, 5% от среднего значения). Это стимулирует активное участие и предотвращает получение выгоды без вклада.

## Позиционирование и дифференциация

FCLC строится на фундаменте существующих фреймворков и консорциумов, предлагая специализированное решение для клинических данных.

*   **Отличие от MELLODDY:** Консорциум MELLODDY фокусируется на молекулярных и биохимических данных для открытия лекарств в фармацевтической индустрии. FCLC специализируется на **клинических наблюдательных данных** (диагнозы, процедуры, исходы) из больниц и реальной клинической практики, решая задачи прогностического моделирования и клинической поддержки принятия решений.
*   **Отличие от FATE / PySyft:** FATE (Federated AI Technology Enabler) и PySyft — это **общего назначения фреймворки** для федеративного обучения. FCLC — это **готовое вертикальное решение**, включающее не только технический стек (который использует компоненты PySyft для SecAgg), но и юридические договоры (DUA), бизнес-модель, клиническую схему данных (OMOP) и управляющую структуру консорциума, что значительно снижает барьер входа для медицинских учреждений.
*   **Отличие от TriNetX:** TriNetX предоставляет централизованную платформу для запросов к агрегированным, обезличенным клиническим данным. FCLC идет дальше, позволяя проводить **распределенное обучение моделей машинного обучения** без передачи даже агрегированных сырых данных, предлагая более строгие гарантии приватности и механизм справедливого распределения выгод от созданных ИИ-активов.

## Правовая и управленческая основа

Успех консорциума зависит от прочного правового и управленческого фундамента.

**1. Юрисдикция и Правовая База:**
*   **Юридическое лицо:** Консорциум будет зарегистрирован как некоммерческое юридическое лицо (ассоциация) в **Грузии**, что обеспечивает нейтральную юрисдикцию для международных участников. Операции будут регулироваться **Законом Грузии о защите персональных данных (PDPL, 2023)**, который гармонизирован с ключевыми принципами GDPR.
*   **Правовое основание для обработки:** Для участников из Европейской экономической зоны обработка специальных категорий данных (медицинских данных) будет основываться на **Статье 9(2)(j) GDPR** («обработка необходимо для целей научных исследований...»), при условии реализации соответствующих гарантий, описанных в данном документе (псевдонимизация, дифференциальная приватность).
*   **Соглашение об использовании данных (DUA):** **Краеугольный документ консорциума.** Стандартизированный шаблон DUA будет первым рабочим продуктом (Deliverable 0). Он будет четко определять:
    *   **Неизменное право собственности:** Все исходные данные остаются исключительной собственностью учреждения-участника.
    *   **Права на производные активы:** Права на глобальные модели, обученные федеративным способом, принадлежат консорциуму как целому, а доступ и коммерческие выгоды распределяются между участниками согласно их вкладу (Shapley Value).
    *   **Разрешенное использование:** Четкое описание целей обучения (например, разработка прогностических моделей для конкретных заболеваний), запрет на попытки де-анонимизации.
    *   **Обязательства по безопасности:** Требования к развертыванию локального узла, аудиту, реагированию на инциденты.
    *   **Выход из консорциума:** Процедуры, включая удаление локального ПО, но сохранение накопленных кредитов вклада для доступа к существующим моделям.

**2. Управление и Операции:**
*   **Управляющий комитет:** Высший руководящий орган, состоящий из избранных представителей от каждой категории участников (клиники, фармацевтические компании, технические партнеры). Комитет утверждает бюджет, стратегию, изменения в DUA и разрешает споры.
*   **Технический операционный комитет:** Отвечает за техническую roadmap, валидацию протоколов безопасности (например, SecAgg+), обслуживание оркестратора и аудит системы оценки вклада.
*   **Механизм разрешения споров:** Формализованный процесс эскалации: 1) Технический аудит, 2) Рассмотрение Управляющим комитетом, 3) При необходимости — арбитраж в соответствии с юрисдикцией Грузии.
*   **Аудит и прозрачность:** Ежеквартальные отчеты о распределении вклада (Shapley Value), использовании бюджета приватности (ε_total), и общих показателях производительности модели. Внешний аудит безопасности и процессов каждые два года.

## Модель дохода

Консорциум использует модель **SaaS-лицензирования с уровнями (tiers)**, напрямую привязанными к системе оценки вклада (Shapley Value). Доход формируется из двух потоков:

1.  **Лицензионные сборы с участников:** Доступ к глобальным моделям и платформе предоставляется по подписке. **Базовая стандартная лицензионная ставка** устанавливается Управляющим комитетом на годовой основе. Размер скидки на эту ставку определяется накопленным кредитом вклада участника (например, 0-25% вклада = 0% скидка, 25-50% = 25% скидка, 50-75% = 50% скидка, >75% = 75% скидка). Это напрямую стимулирует качественный вклад.
2.  **Коммерческая выручка от сторонних лицензий:** Если консорциум лицензирует обученные модели внешним сторонам (например, разработчикам медицинского ПО), **70%** чистой выручки распределяется между участниками-учредителями пропорционально их накопленному вкладу (Shapley Value). **30%** реинвестируется в операционные расходы и развитие платформы.

Эта модель обеспечивает устойчивость, справедливость и прямую связь между вкладом данных и финансовой выгодой.

## План реализации (MVP)

Общий реалистичный срок для полноценного MVP: **12-18 месяцев**.

**Фаза 0: Правовая и организационная основа (3 месяца)**
*   **Deliverable 0.1:** Окончательный шаблон Соглашения об использовании данных (DUA).
*   **Deliverable 0.2:** Учредительные документы консорциума в Грузии, формирование Управляющего комитета.
*   **Deliverable 0.3:** Подписание первых 3-5 пилотными участниками DUA и получение локальных одобрений IRB/этических комитетов (ожидаемый срок: 3-6 месяцев, выполняется параллельно с Фазой 1).

**Фаза 1: Техническая разработка и валидация (6 месяцев)**
*   **Deliverable 1.1:** Развертывание тестового оркестратора и разработка пакета локального узла (адаптер OMOP, обезличивание, клиент FL).
*   **Deliverable 1.2:** Интеграция и валидация протоколов SecAgg+ (PySyft) и дифференциальной приватности на синтетических данных.
*   **Deliverable 1.3:** Реализация и тестирование конвейера оценки вклада (Fed-SV) и алгоритмов робастной агрегации (FedProx, Krum).

**Фаза 2: Пилот с реальными данными (6 месяцев)**
*   **Deliverable 2.1:** Развертывание локальных узлов у 3-5 пилотных участников на их изолированных тестовых средах.
*   **Deliverable 2.2:** Обучение первой федеративной модели на реальных, но ограниченных клинических данных (например, прогнозирование госпитализации при ХСН).
*   **Deliverable 2.3:** Полный аудит процесса: безопасность, приватность, корректность расчета вклада, производительность модели. Подготовка к масштабированию.

## Заключение

FCLC предлагает технически зрелую, юридически обоснованную и коммерчески устойчивую модель для раскрытия потенциала клинических данных с помощью федеративного обучения. Решение напрямую устраняет ключевые барьеры — опасения по поводу приватности и отсутствие справедливых стимулов — через архитектурные гарантии приватности, основанную на теории игр метрику вклада и четкую правовую структуру. Это создает необходимые условия для формирования жизнеспособного кооператива ведущих медицинских учреждений.
